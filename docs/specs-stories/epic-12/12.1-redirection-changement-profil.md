# Story 12.1 — Redirection après changement de profil

**Titre** : Changer de profil doit quitter l'écran "Profil & paramètres" et rediriger vers : **Accueil** si application déjà débloquée, sinon **Checklist**

**Priorité** : P3

---

## Objectif fonctionnel

Quand un utilisateur change de profil, il faut le rediriger tout de suite vers le bon écran. Si l'application n'est pas encore débloquée alors l'écran le plus adapté est l'écran de checklist. Si l'application est débloquée, l'écran d'accueil est l'écran le plus adapté.

## Règles métier exactes

- Si le profil ciblé a un mot de passe défini : que je sois propriétaire ou non, si je change d'utilisateur et que le mot de passe (PIN propre à ce profil) saisi est bon, alors je suis redirigé vers l'écran d'accueil si l'application (family-wide) est débloquée, sinon vers l'écran de checklist.
- Si le profil ciblé n'a pas de mot de passe défini : aucune saisie n'est demandée, le changement de profil s'applique directement et la même règle de redirection s'applique (accueil si débloqué, checklist sinon).

## Cas limites

- Mot de passe incorrect → message d'erreur affiché, l'utilisateur reste sur la popup de saisie, aucune redirection. Pas de limite de tentatives, aucun verrouillage.
- L'utilisateur peut annuler le changement de profil à tout moment depuis la popup → retour à l'écran "Profil & paramètres", aucun changement de profil n'est effectué.
- Profil sans mot de passe défini → pas de popup de saisie du tout, redirection immédiate selon l'état de déverrouillage.
- Changer vers le profil déjà actif (avec bon mot de passe, ou sans mot de passe) → la redirection s'applique quand même (comportement idempotent, pas de cas spécial).
- Juste après un switch réussi, un refresh de page doit atterrir directement sur le bon écran (dépend du fix de la story 1.2 — écran de chargement atomique — à ne pas re-casser ici).

## Critères d'acceptation testables

1. Profil avec mot de passe + saisie correcte + application débloquée → redirection vers Accueil.
2. Profil avec mot de passe + saisie correcte + application non débloquée → redirection vers Checklist.
3. Profil avec mot de passe + saisie incorrecte → message d'erreur affiché, reste sur la popup de saisie, aucune redirection, aucune limite de tentatives.
4. Popup de saisie affichée + clic sur "Annuler" → retour à l'écran "Profil & paramètres", profil actif inchangé.
5. Profil sans mot de passe défini → aucune popup affichée, redirection immédiate (Accueil si débloqué, Checklist sinon).
6. La redirection est déterminée par l'état de déverrouillage **family-wide** (issu des stories 1.3/1.6), pas par un état local au profil ou au navigateur.

## Hors périmètre

- Définition ou modification du mot de passe/PIN d'un profil (autre story).
- Le code propriétaire family-wide lui-même (déjà couvert par 1.3/1.6).
- Comportement multi-onglets/multi-fenêtres simultanés sur le même appareil.

## Validation attendue

Tests unitaires, intégration, e2e.

## Contraintes UX

- La redirection doit être immédiate après validation du mot de passe (ou immédiatement si pas de mot de passe requis), sans écran intermédiaire vide (cohérent avec le fix de chargement atomique de la story 1.2).
- Le message d'erreur en cas de mauvais mot de passe s'affiche sur la popup elle-même, sans la fermer.
- Éviter tout flash visuel de l'ancien profil/écran avant que la redirection ne soit effective.
