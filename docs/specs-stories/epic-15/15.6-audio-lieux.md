# Story 15.6 (ex-5.6) — Correction du chemin des fichiers audio et gestion de leur absence

**Titre** : Corriger le chemin de référence des fichiers audio (`/audio/places/{lieu}.mp3`) et garantir une gestion propre de leur absence en attendant les fichiers définitifs

**Priorité** : P1

---

## Contexte

Les fichiers audio ne sont pas encore prêts (confirmé). Cette story ne porte donc pas sur la production du contenu audio, mais sur la **préparation technique** : corriger le chemin de référence pour qu'il corresponde à la structure de dossiers cible, et s'assurer que l'absence actuelle de fichiers est gérée proprement, sans bug, en attendant que tu fournisses les mp3.

Structure de dossiers cible (déjà définie) :
```
public/
  audio/
    places/
      istanbul.mp3
      cappadoce.mp3
      pamukkale.mp3
      ephese.mp3
      izmir.mp3
      bursa.mp3
```

Le code référence actuellement les fichiers via `/audio/{lieu}.mp3` (chemin plat), ce qui ne correspond pas à cette structure (`/audio/places/{lieu}.mp3`). Sans correction, même une fois les fichiers ajoutés au bon endroit, ils ne seraient pas trouvés par l'application.

## Objectif fonctionnel

Corriger le chemin de référence vers les fichiers audio pour qu'il corresponde à la structure de dossiers définie, et garantir que l'absence de fichier (situation actuelle) est gérée proprement côté interface, sans erreur technique visible.

## Règles métier exactes

- Le code doit référencer les fichiers audio via `/audio/places/{lieu}.mp3` (et non plus `/audio/{lieu}.mp3`).
- Tant qu'un fichier audio n'existe pas physiquement dans `public/audio/places/`, l'application affiche un message clair ("Audio indisponible pour le moment", déjà en place visuellement) sans erreur technique ni blocage de la page.
- Dès qu'un fichier audio est ajouté dans le bon dossier avec le bon nom, il doit être automatiquement lu par l'application, sans modification de code supplémentaire.

## Cas limites

- Fichier présent mais corrompu ou dans un format invalide → le même message de fallback doit s'afficher, plutôt qu'une erreur technique brute ou un lecteur audio cassé.
- Nom de fichier qui ne correspond pas exactement à l'identifiant du lieu (casse, accents, tiret vs underscore) → à documenter clairement pour toi comme convention de nommage, pour éviter ce problème quand tu ajouteras les vrais fichiers plus tard.
- ❓ **À vérifier par le Dev** : est-ce que tous les lieux de `places.ts` ont une entrée `audioSrc` prévue, ou certains lieux n'ont pas de contenu audio du tout ? Si un lieu n'a jamais eu vocation à avoir un audio, il ne doit pas afficher le message "Audio indisponible" (qui laisserait penser à un bug) mais plutôt ne rien afficher du tout à cet endroit.

## Critères d'acceptation testables

1. Le code référence les fichiers audio via `/audio/places/{lieu}.mp3`, conforme à la structure de dossiers cible.
2. En l'absence du fichier (situation actuelle), le message "Audio indisponible pour le moment" s'affiche proprement sur chaque lieu concerné, sans erreur technique visible dans la console ni écran cassé.
3. Ajouter manuellement un fichier mp3 de test dans le bon dossier avec le bon nom fait apparaître un lecteur audio fonctionnel, sans modification de code.
4. Un fichier audio corrompu ou invalide déclenche le même message de fallback qu'un fichier absent, pas une erreur brute.
5. Un lieu qui n'a jamais eu vocation à avoir un audio (si ce cas existe, voir cas limite ci-dessus) n'affiche pas le message "Audio indisponible" à tort.

## Hors périmètre

- Production, enregistrement ou génération des fichiers audio eux-mêmes (à fournir par toi plus tard, story de suivi si besoin).
- Choix du contenu ou du texte lu dans les audios.
- Interface de gestion des fichiers audio depuis l'application.

## Priorité

P2

## Validation attendue

Tests unitaires (gestion du cas fichier absent/corrompu), test manuel avec un fichier mp3 de test ajouté temporairement pour valider le bon fonctionnement une fois le fichier réellement présent.

## Contraintes UX

- Le message de fallback reste discret et cohérent visuellement avec le design existant (déjà le cas d'après la capture d'écran) — pas de changement visuel attendu, sauf si un bug d'affichage est découvert en testant le cas du fichier corrompu.
