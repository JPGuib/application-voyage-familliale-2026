# Story 15.5 (ex-5.5) — Rubrique "Guide de visite" détaillée pour les monuments visités

**Titre** : Ajouter une rubrique "Guide de visite" détaillée (texte + photos) pour les monuments visités

**Priorité** : P1

---

## Objectif fonctionnel

Permettre d'afficher, pour les lieux qui sont des monuments réellement visités, un contenu approfondi (texte structuré en sections/sous-sections + photos), rédigé par toi dans un document Word placé dans `.docs/`. Ce contenu peut représenter plusieurs pages, ce qui a une conséquence directe sur l'architecture (voir décision ci-dessous).

## Décision d'architecture : nouvel écran dédié (recommandation)

Tu as toi-même soulevé la question ("nouvel écran ou pas ?") — voici ma recommandation, avec la justification :

**→ Nouvel écran dédié**, accessible via une carte/bouton "Voir le guide de visite complet" affiché **en dessous de la section "Le savais-tu ?"** sur l'écran de détail actuel du lieu (visible sur ta capture).

**Pourquoi pas un affichage inline sous "Le savais-tu"** :
- Le contenu peut faire "plusieurs pages avec des sous-sections" — l'intégrer directement rendrait l'écran de détail actuel excessivement long, y compris pour les lieux qui n'ont pas ce contenu détaillé.
- Un écran dédié permet d'ajouter une navigation interne propre aux sous-sections (ex : sommaire cliquable en haut de l'écran), ce qui serait très inconfortable à faire au milieu d'un écran déjà chargé (galerie, histoire, savais-tu, etc.).
- Le contenu ne se charge que si l'utilisateur clique dessus (meilleure performance), plutôt que d'alourdir systématiquement l'écran de détail pour tout le monde.

**Affichage conditionnel** : cette section (carte + lien) n'apparaît **que pour les lieux qui ont effectivement un contenu de guide de visite fourni** (pas tous les lieux) — la présence du contenu détermine l'affichage, pas un indicateur "monument" séparé à maintenir manuellement.

## Règles métier exactes

- Une carte "Voir le guide de visite complet" (libellé à ajuster selon le style de l'app) apparaît sous "Le savais-tu ?" uniquement si un contenu de guide de visite existe pour ce lieu.
- Cliquer sur cette carte ouvre un nouvel écran dédié, affichant le contenu structuré (sections avec titres, sous-sections, texte, photos) tel que rédigé dans le document Word source.
- Le contenu source est un document Word placé par toi dans `.docs/visites-guidees/{lieu}.docx` (nommage : identifiant du lieu utilisé dans `places.ts`).
- Un **template Word** est fourni (voir fichier joint `modele-guide-de-visite.docx`) — structure libre, tu peux ajouter/retirer des sections et sous-sections à volonté, seuls les styles "Titre 1"/"Titre 2" de Word doivent être utilisés pour que la conversion fonctionne correctement.
- Les photos sont insérées **directement dans le document Word**, à l'endroit exact où tu veux qu'elles apparaissent — pas besoin de les nommer ou de les fournir séparément, elles seront extraites automatiquement du document lors de la conversion.

## Point technique pour le Dev/Architect (pipeline de conversion)

Le document Word n'est pas directement exploitable par l'application React — une étape de conversion est nécessaire :
- Utiliser une bibliothèque de conversion docx → HTML (ex : `mammoth`, qui préserve les styles de titre et extrait automatiquement les images embarquées) lors d'un script de build, pour transformer chaque `.docx` de `.docs/visites-guidees/` en un contenu exploitable par l'écran dédié (HTML structuré ou JSON avec sections/sous-sections + chemins d'images extraites).
- Les images extraites doivent être sauvegardées dans le dossier de photos local déjà défini pour ce lieu (`public/images/places/{lieu}/`, cf. story 15.1), pour rester cohérent avec la stratégie d'hébergement local des images.
- Cette conversion peut se faire au moment du build (recommandé, pas de dépendance runtime) ou via un script à lancer manuellement après chaque ajout/modification de document — à trancher par le Dev selon la complexité d'intégration avec Vite.

## Cas limites

- Lieu sans document Word fourni → aucune carte "Guide de visite" n'apparaît, comportement actuel inchangé.
- Document Word présent mais mal structuré (pas de styles Titre utilisés) → le Dev doit prévoir un comportement de repli raisonnable (ex : afficher le texte brut sans structure plutôt que de planter), et t'alerter si un document ne peut pas être converti correctement.
- Document très long (plusieurs pages) → l'écran dédié doit rester confortable à parcourir (scroll simple, éventuellement un sommaire en haut avec des liens d'ancrage vers chaque section).
- Photo insérée dans le Word mais dans un format non standard ou trop lourd → à compresser/convertir automatiquement lors de l'extraction, selon les mêmes specs de taille que la story 15.1 (hero/vignette).

## Critères d'acceptation testables

1. Un document Word structuré (basé sur le template fourni) placé dans `.docs/visites-guidees/{lieu}.docx` génère un contenu affichable dans un nouvel écran dédié.
2. Une carte "Voir le guide de visite complet" apparaît sous "Le savais-tu ?" uniquement pour les lieux ayant un document fourni.
3. Cliquer sur cette carte ouvre l'écran dédié avec le contenu structuré (titres, sous-titres, texte, photos) fidèle au document Word.
4. Les photos insérées dans le document Word apparaissent correctement dans l'écran dédié, à l'emplacement où elles ont été insérées dans le document source.
5. Un lieu sans document fourni n'affiche aucune carte ni lien vers un guide de visite (non-régression, comportement actuel préservé).
6. Le contenu reste lisible et bien structuré sur mobile (Android/iOS) comme sur ordinateur.

## Hors périmètre

- Rédaction du contenu lui-même (texte, choix des photos) — fournie par toi via Word.
- Interface d'édition de ce contenu directement depuis l'application.
- Traduction ou gestion multi-langue du contenu.

## Priorité

P1

## Validation attendue

Tests unitaires (conversion docx → contenu structuré), tests d'intégration (affichage de l'écran dédié), test manuel avec un document Word réel basé sur le template.

## Contraintes UX

- La carte d'accès au guide de visite doit être visuellement engageante mais ne pas dominer l'écran de détail actuel (elle reste un complément, pas l'élément principal).
- L'écran dédié doit rester cohérent visuellement avec le reste de l'application (même charte graphique, mêmes styles de titres que le reste du Guide).
- Prévoir un bouton de retour clair vers l'écran de détail du lieu depuis l'écran dédié.

`modele-guide-de-visite.docx` — template Word avec instructions intégrées, structure d'exemple (Histoire / Ce qu'il faut voir / Anecdotes), à dupliquer et adapter pour chaque monument.
