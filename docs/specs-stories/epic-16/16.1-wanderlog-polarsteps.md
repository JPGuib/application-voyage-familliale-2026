# Story 16.1 (fusion ex-6.1, ex-6.2, ex-6.3) — Ouverture des liens Wanderlog et Polarsteps

**Titre** : Boutons Wanderlog et Polarsteps : ouvrir l'application native si installée (mobile), sinon le site web — dans tous les cas, dans un nouvel onglet/une nouvelle fenêtre

**Priorité** : P3

---

## Note de fusion

Cette story fusionne les trois items initiaux du backlog (6.1 Wanderlog, 6.2 Polarsteps, 6.3 nouvel onglet) car ils partagent le même mécanisme technique : ouverture conditionnelle app-ou-web, pour les deux boutons, avec le même comportement de nouvel onglet. Les traiter séparément aurait dupliqué le même travail deux fois.

## Objectif fonctionnel

Depuis l'écran d'accueil, les boutons **Wanderlog** et **Polarsteps** doivent :
- ouvrir l'application native correspondante si elle est installée sur l'appareil (smartphone Android/iOS),
- sinon rediriger vers le site web correspondant,
- dans tous les cas, sans jamais bloquer ou remplacer l'écran de l'application de voyage en cours (nouvel onglet/nouvelle fenêtre).

## Première tâche : investigation technique (avant tout développement)

Il faut d'abord déterminer si Wanderlog et Polarsteps ont configuré des **Universal Links** (iOS) / **App Links** (Android) sur leurs domaines respectifs — cela détermine complètement l'approche technique à suivre.

**Comment vérifier** :
- Envoyer un lien `https://wanderlog.com/...` (ou équivalent Polarsteps) via une messagerie (SMS, notes) sur un téléphone où l'application est installée, et observer si le lien ouvre directement l'app ou le navigateur.
- Vérifier la présence des fichiers de vérification `/.well-known/apple-app-site-association` et `/.well-known/assetlinks.json` sur les domaines wanderlog.com et polarsteps.com.

**Deux scénarios possibles selon le résultat** :
- **Scénario A — Universal Links/App Links déjà configurés par Wanderlog/Polarsteps** : il suffit d'ouvrir le lien https standard dans un nouvel onglet ; le système d'exploitation ouvre automatiquement l'application si elle est installée, sans code spécifique à écrire. Solution la plus simple et la plus fiable — à privilégier si disponible.
- **Scénario B — pas de Universal Links/App Links configurés** : il faut alors utiliser un schéma d'URL personnalisé propre à chaque application (ex : `wanderlog://`) avec un mécanisme de repli (tenter le schéma personnalisé, puis rediriger automatiquement vers l'URL web après un court délai si l'app ne s'active pas). Solution plus complexe et moins fiable, à n'implémenter que si le scénario A ne fonctionne pas.

## Règles métier exactes

- Sur smartphone (Android/iOS), cliquer sur le bouton Wanderlog ouvre l'application Wanderlog si installée, sinon le site web.
- Idem pour Polarsteps.
- Sur ordinateur, le comportement actuel (ouverture du site web) est conservé, mais dans un **nouvel onglet**.
- Dans tous les cas (app ou web), l'ouverture ne doit jamais remplacer ni bloquer l'écran de l'application de voyage en cours.

## Cas limites

- Universal Links désactivés par l'utilisateur dans les réglages système (possible sur iOS/Android) → repli naturel vers le site web, comportement acceptable.
- Navigateur intégré (in-app browser d'une autre app, ex. réseau social) → les Universal Links peuvent être moins fiables dans ce contexte ; le repli vers le web reste un comportement acceptable.
- Si le scénario B est nécessaire : le délai de détection doit être court (1 à 2 secondes maximum) pour ne pas donner l'impression que l'app est bloquée, tout en laissant le temps au système de basculer vers l'app native si elle doit s'ouvrir.

## Critères d'acceptation testables

1. Sur Android avec Wanderlog installé, cliquer sur le bouton ouvre l'application native.
2. Sur Android sans Wanderlog installé, cliquer sur le bouton ouvre le site web dans un nouvel onglet.
3. Mêmes comportements sur iOS.
4. Mêmes comportements pour Polarsteps (Android et iOS, avec et sans l'app installée).
5. Sur ordinateur, les boutons ouvrent le site web dans un nouvel onglet, sans remplacer l'onglet de l'application de voyage.
6. Dans tous les cas testés, l'application de voyage reste immédiatement accessible après le clic (pas de blocage, pas de perte de contexte).

## Hors périmètre

- Toute intégration technique poussée avec les API de Wanderlog/Polarsteps (partage/synchronisation de données) — uniquement un lien de redirection.
- Le contenu ou fonctionnement propre des applications Wanderlog et Polarsteps, hors de notre contrôle.

## Priorité

P1 (reprend la priorité la plus haute des trois stories fusionnées, celle du bug bloquant initial 6.3).

## Validation attendue

Tests manuels sur appareils réels (Android et iOS, avec et sans les apps installées), test sur ordinateur (Chrome, Edge, Safari).

## Contraintes UX

- Aucune popup ni message d'erreur ne doit apparaître si l'app n'est pas installée — juste une redirection fluide vers le web.
- Le bouton doit rester visuellement réactif (état "au clic") le temps que la redirection s'exécute, pour éviter une impression de bouton qui ne répond pas.
