# Story 18.8 — Retrait de la "Phrase de récupération" et affichage du code propriétaire actuel

**Titre** : La section "Phrase de récupération" (code propriétaire) ne sert plus à rien — la remplacer par la possibilité d'afficher le code propriétaire actuellement configuré

**Priorité** : P3

---

## Contexte (écran réel)

L'écran Paramètres du propriétaire contient trois sections distinctes :
1. **"Récupération du mot de passe profil"** — question/réponse personnelle, propre à chaque profil (couverte par 18.4/18.5/18.6, **inchangée**).
2. **"Code propriétaire"** — permet de remplacer le code actuel via un champ de saisie + "Afficher le code saisi" (révèle ce qu'on est en train de taper, pas le code déjà configuré) + bouton "Mettre à jour le code".
3. **"Phrase de récupération"** — mécanisme séparé, spécifique au propriétaire, jugé obsolète et à retirer.

**Le problème soulevé** : si on retire la "Phrase de récupération" sans rien la remplacer, le propriétaire n'a plus aucun moyen de retrouver son code s'il l'oublie — la section "Code propriétaire" actuelle ne permet de voir que ce qu'on tape pour le *remplacer*, jamais le code *déjà configuré*.

**Solution retenue** : plutôt qu'un mécanisme de récupération, le propriétaire doit pouvoir **consulter directement le code actuellement configuré**, masqué par défaut et révélable à la demande — ce qui rend la phrase de récupération inutile.

## Objectif fonctionnel

Retirer l'affichage de la section "Phrase de récupération", et compléter la section "Code propriétaire" pour permettre au propriétaire de voir le code actuellement configuré (masqué par défaut, révélable à la demande), indépendamment du champ servant à le remplacer.

## Règles métier exactes

- La section **"Phrase de récupération"** n'est plus affichée dans l'écran Paramètres. Pour limiter le risque de régression, seul l'**affichage** est retiré — les données déjà stockées ne sont pas nécessairement supprimées à ce stade (à la discrétion du Dev, tant que rien ne casse ailleurs).
- La section **"Code propriétaire"** est complétée pour afficher le **code actuellement configuré**, masqué par défaut (ex : ●●●●), avec un lien "Afficher le code actuel" permettant de le révéler à la demande.
- Cet affichage du code actuel est **indépendant** du champ de saisie d'un nouveau code : celui-ci garde son propre "Afficher le code saisi" pour ce qui est en train d'être tapé, sans confusion entre les deux.
- Seul le propriétaire a accès à cette section (déjà le cas aujourd'hui).

## Cas limites

- Le code actuel révélé doit se re-masquer automatiquement si l'utilisateur quitte l'écran puis y revient (cohérent avec le comportement défini en 18.5).
- Avant de retirer la "Phrase de récupération", vérifier qu'aucune autre partie du code ne s'appuie encore dessus (ex : un flux de récupération qui l'utiliserait en arrière-plan) — si un tel usage existe, le signaler avant suppression complète, pour ne pas casser silencieusement une fonctionnalité encore active.

## Critères d'acceptation testables

1. La section "Phrase de récupération" n'apparaît plus dans l'écran Paramètres du propriétaire.
2. La section "Code propriétaire" affiche le code actuellement configuré, masqué par défaut.
3. Un lien "Afficher le code actuel" révèle ce code à la demande.
4. Le code révélé se re-masque automatiquement en quittant l'écran.
5. Le champ de saisie d'un nouveau code et son "Afficher le code saisi" restent inchangés et indépendants de l'affichage du code actuel.
6. Aucune régression sur les sections "Récupération du mot de passe profil" et "Mot de passe du profil".

## Hors périmètre

- Suppression des données déjà stockées liées à l'ancienne "phrase de récupération" (nettoyage technique éventuel, à traiter dans une story séparée si nécessaire).
- Toute modification du mécanisme de récupération du mot de passe profil (18.4/18.5/18.6), qui reste inchangé.

## Priorité

P3

## Validation attendue

Tests unitaires, intégration, e2e (retrait de la section, affichage/masquage/révélation du code actuel, non-régression sur les autres sections).

## Contraintes UX

- Cohérence visuelle avec le mécanisme déjà utilisé pour "Afficher la phrase saisie"/"Afficher le code saisi" (même style de lien).
- Le code actuel révélé doit être visuellement bien distinct du champ de saisie d'un nouveau code, pour éviter toute confusion entre "ce qui est configuré actuellement" et "ce que je suis en train de taper pour le remplacer".
