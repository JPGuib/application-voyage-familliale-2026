# Story 18.1 — Reverrouillage de l'application par le propriétaire

**Titre** : Pouvoir rebloquer l'application (en plus de la débloquer) — pendant le blocage, seule la Checklist reste visible

**Priorité** : P0

---

## Objectif fonctionnel

Permettre au propriétaire de reverrouiller l'application à tout moment (en plus de pouvoir la débloquer, comme aujourd'hui). Une fois reverrouillée, tous les profils — y compris le propriétaire dans cette story — ne voient plus que l'écran Checklist, jusqu'au prochain déblocage.

## Règles métier exactes

- Le propriétaire peut déclencher un **reverrouillage** de l'application à tout moment, via une action dédiée (ex : dans les paramètres).
- Pour bloquer **ou** débloquer, le propriétaire doit ressaisir son code à chaque fois — même mécanisme que le déverrouillage déjà existant (cf. stories 1.6 / 12.1).
- Une fois l'application reverrouillée : **tous les profils, y compris le propriétaire**, ne voient plus que l'écran Checklist ; les autres écrans (Accueil, Guide, Jeu, Conseils, Résultats) sont inaccessibles jusqu'au prochain déblocage.
- Ce comportement est cohérent avec l'état "avant premier déverrouillage" déjà existant dans l'application (phase "before").
- Le code saisi (bon ou mauvais) suit les mêmes règles déjà définies pour le déverrouillage : mauvais code → message d'erreur, pas de limite de tentatives (cohérent avec 12.1).
- **Aucune confirmation supplémentaire** n'est nécessaire : l'action est immédiate dès que le bon code est saisi.

## Cas limites

- Reverrouillage déclenché pendant qu'un autre profil utilise un autre écran (Guide, Jeu...) → cet utilisateur est automatiquement redirigé vers Checklist dès que l'état reverrouillé lui parvient (propagation family-wide, cf. architecture des stories 1.3/1.4), sans nécessiter de rafraîchissement manuel.
- Reverrouiller puis déverrouiller immédiatement avec le bon code → comportement identique à un cycle normal, pas de cas particulier.
- Le propriétaire tente de reverrouiller alors que l'application est déjà verrouillée (ex : deux onglets ouverts) → aucune action supplémentaire, l'état reste simplement verrouillé, sans erreur.

## Critères d'acceptation testables

1. Depuis les paramètres, le propriétaire peut déclencher un reverrouillage après avoir saisi correctement son code.
2. Après reverrouillage, tous les profils de la famille (y compris le propriétaire) ne voient plus que l'écran Checklist, sur tous leurs appareils.
3. Un profil actif sur un autre écran au moment du reverrouillage est automatiquement redirigé vers Checklist, sans action de sa part.
4. Un mauvais code lors de la tentative de reverrouillage affiche un message d'erreur, sans limite de tentatives (cohérent avec 12.1).
5. Depuis l'état reverrouillé, ressaisir le bon code permet de débloquer à nouveau l'application normalement (cycle complet fonctionnel).
6. Aucune popup de confirmation supplémentaire n'apparaît avant le reverrouillage effectif.

## Hors périmètre

- **Exception d'accès pour le propriétaire pendant le reverrouillage** : dans cette story, le propriétaire voit uniquement la Checklist comme tout le monde. Une **story future** permettra au propriétaire de conserver un accès complet même reverrouillé — non traité ici.
- Toute notification poussée informant les autres profils du reverrouillage (juste une redirection silencieuse vers Checklist, pas de message explicatif).

## Priorité

P2

## Validation attendue

Tests unitaires, intégration, e2e (cycle verrouillage/déverrouillage complet, propagation multi-profils et multi-appareils).

## Contraintes UX

- Le bouton/action de reverrouillage doit être clairement accessible dans les paramètres, sans être trop discret, même en l'absence de confirmation supplémentaire.
- La transition vers Checklist pour les autres profils doit être fluide, sans écran blanc intermédiaire (cohérent avec les fixes des stories 1.1/1.2).

---

## Note de portée future

Une prochaine story permettra au propriétaire de conserver un accès complet à l'application même lorsqu'elle est reverrouillée pour les autres utilisateurs. Ce n'est **pas** implémenté dans cette story-ci — à concevoir séparément, avec un impact probable sur le modèle family-wide actuel puisqu'il faudra alors distinguer un état "verrouillé pour les autres, ouvert pour le propriétaire".
