# Story 18.3 — Suppression de son propre profil

**Titre** : Un utilisateur doit pouvoir supprimer son profil et toutes les données le concernant (sans impact sur les autres utilisateurs) — le propriétaire ne peut jamais être supprimé

**Priorité** : P0

---

## Objectif fonctionnel

Permettre à un utilisateur (profil non-propriétaire) de supprimer définitivement son propre profil ainsi que toutes les données qui lui sont associées, depuis l'écran Paramètres, sans impacter les autres profils de la famille. Le propriétaire ne peut jamais être supprimé.

## Règles métier exactes

- L'option de suppression de profil est disponible dans l'écran Paramètres, pour **tout profil sauf le propriétaire**.
- Le bouton/option de suppression n'est **jamais disponible** pour le profil propriétaire (masqué ou absent).
- Avant suppression : un **popup d'avertissement** informe clairement que l'action est irréversible et supprimera toutes les données du profil.
- Confirmation de la suppression :
  - Si le profil a un mot de passe défini → confirmation via ce mot de passe, ou via le mécanisme **existant** de question de récupération (en cas de mot de passe oublié).
  - Si le profil n'a **pas** de mot de passe défini → le popup d'avertissement seul suffit comme confirmation, aucun mot de passe n'est demandé.
- La suppression retire : le profil lui-même, sa checklist, son historique de jeu/résultats, et ses catégories personnalisées privées créées par lui (cf. story 12.3).
- La suppression **ne doit affecter aucune autre donnée** : ni celles des autres profils, ni les catégories/items partagés créés par le propriétaire, ni la table jour→destination, ni aucune donnée family-wide.
- Après suppression réussie, l'utilisateur est redirigé vers l'écran de **sélection de profil**.

## Cas limites

- Tentative de suppression du profil propriétaire → impossible, option non disponible pour lui.
- Mot de passe incorrect saisi pour confirmer → message d'erreur, suppression annulée, profil intact (aucune donnée supprimée).
- Utilisation du mécanisme de question de récupération pour confirmer (mot de passe oublié) → suit le comportement déjà existant de ce mécanisme, non modifié ici.
- Suppression du dernier profil non-propriétaire de la famille → l'application reste fonctionnelle avec au minimum le profil propriétaire restant ; l'écran de sélection de profil n'affiche alors plus que lui.
- Profil supprimé alors qu'il est actif simultanément sur un autre appareil → cet autre appareil doit être redirigé de façon cohérente (vers la sélection de profil) plutôt que de continuer à fonctionner sur un profil qui n'existe plus.

## Critères d'acceptation testables

1. Dans l'écran Paramètres, un profil non-propriétaire dispose d'une option "Supprimer mon profil".
2. Le profil propriétaire ne dispose jamais de cette option.
3. Cliquer sur "Supprimer mon profil" affiche un popup d'avertissement explicite sur l'irréversibilité de l'action.
4. Si le profil a un mot de passe, celui-ci (ou la question de récupération existante) est demandé avant la suppression effective.
5. Si le profil n'a pas de mot de passe, le popup d'avertissement seul suffit pour confirmer.
6. Après confirmation, le profil, sa checklist, son historique de jeu et ses catégories privées sont définitivement supprimés.
7. Les autres profils de la famille (données, checklist, historique) ne sont pas affectés par cette suppression.
8. Après suppression, l'utilisateur est redirigé vers l'écran de sélection de profil.
9. Un mot de passe incorrect annule la suppression sans supprimer aucune donnée.

## Hors périmètre

- Fonctionnement interne du mécanisme de question de récupération (déjà existant, non modifié ici).
- Suppression du profil propriétaire (explicitement impossible).
- Récupération ou restauration d'un profil supprimé (suppression définitive, pas de corbeille).

## Priorité

P0

## Validation attendue

Tests unitaires, intégration, e2e (suppression réussie, non-régression sur les autres profils, cas avec/sans mot de passe, tentative de suppression du profil propriétaire).

## Contraintes UX

- Le popup d'avertissement doit lister clairement ce qui sera supprimé (checklist, historique, catégories personnalisées) et insister sur le caractère irréversible.
- Le bouton de suppression doit être visuellement distinct (ex : couleur d'alerte) pour éviter tout clic accidentel, tout en restant accessible dans les paramètres.
