# Story 18.2 — Accès complet du propriétaire + indicateur d'état de verrouillage

**Titre** : Le propriétaire voit tous les écrans même si l'application est verrouillée ; ajout d'un badge d'état et d'un bouton unique bloquer/débloquer dans les paramètres

**Priorité** : P3

---

## Objectif fonctionnel

Le propriétaire — et uniquement lui — doit pouvoir accéder à tous les écrans de l'application à tout moment, que celle-ci soit verrouillée ou non pour les autres utilisateurs. Il reste le seul profil capable de bloquer/débloquer l'application (cf. story 18.1). Pour qu'il sache toujours dans quel état se trouve l'application, un badge d'état et un bouton d'action unique (dont le libellé change selon l'état) sont ajoutés dans les paramètres.

## Règles métier exactes

- Quel que soit l'état de verrouillage de l'application (verrouillée ou non pour les autres profils), le **propriétaire** a accès à tous les écrans (Accueil, Guide, Jeu, Conseils, Résultats, Checklist) sans restriction.
- Les profils non-propriétaires restent soumis à la règle de la story 18.1 : uniquement Checklist visible quand l'application est verrouillée.
- Seul le propriétaire peut déclencher un blocage ou un déblocage, en ressaisissant son code à chaque fois (cohérent avec 18.1/1.6) — pas de changement sur ce point.
- Dans l'écran Paramètres du propriétaire :
  - Un **badge d'état** indique clairement si l'application est actuellement "Verrouillée" ou "Débloquée".
  - Un **bouton unique** permet d'agir, dont le libellé s'adapte à l'état courant :
    - Application débloquée → le bouton affiche **"Bloquer l'application"**.
    - Application verrouillée → le bouton affiche **"Débloquer l'application"**.
  - Cliquer sur ce bouton déclenche la saisie du code (comme aujourd'hui), et une fois validé, bascule l'état et met à jour immédiatement le badge et le libellé du bouton.

## Cas limites

- Le propriétaire navigue vers n'importe quel écran pendant que l'application est verrouillée pour les autres → accès normal, sans redirection vers Checklist (contrairement aux autres profils).
- Le badge et le bouton ne sont visibles que dans les paramètres du profil **propriétaire** — un utilisateur normal ne voit ni le badge ni ce bouton dans son propre écran de paramètres.
- Bascule de l'état (verrouillage ↔ déverrouillage) déclenchée par le propriétaire sur un appareil → le badge se met à jour sur tous ses autres appareils également (cohérent avec le modèle family-wide, cf. story 1.3), sans qu'il ait besoin de rafraîchir manuellement.
- Mauvais code saisi lors du clic sur le bouton → message d'erreur, aucun changement d'état, bouton et badge inchangés (cohérent avec 12.1/18.1).

## Critères d'acceptation testables

1. Le propriétaire accède à tous les écrans de l'application sans restriction, que l'application soit verrouillée ou non pour les autres profils.
2. Un utilisateur non-propriétaire reste limité à Checklist quand l'application est verrouillée (non-régression de la story 18.1).
3. Dans les paramètres du propriétaire, un badge affiche clairement l'état courant ("Verrouillée" ou "Débloquée").
4. Le bouton d'action affiche "Bloquer l'application" quand l'app est débloquée, et "Débloquer l'application" quand elle est verrouillée.
5. Cliquer sur le bouton demande le code ; une fois validé, l'état bascule et le badge/bouton se mettent à jour immédiatement.
6. Le badge et le bouton ne sont visibles que dans les paramètres du profil propriétaire, pas dans ceux d'un profil normal.
7. Le badge reflète le même état sur tous les appareils du propriétaire, sans rafraîchissement manuel nécessaire.

## Hors périmètre

- Affichage d'un indicateur d'état de verrouillage ailleurs que dans l'écran Paramètres (ex : bandeau permanent sur toutes les pages) — non demandé ici, à envisager comme amélioration future si souhaité.
- Toute notification poussée vers les autres profils lors du changement d'état (déjà hors périmètre depuis la story 18.1).

## Priorité

P2

## Validation attendue

Tests unitaires, intégration, e2e (accès complet du propriétaire pendant verrouillage, mise à jour du badge/bouton, non-régression pour les profils normaux).

## Contraintes UX

- Le badge doit être visuellement clair et rapide à interpréter (ex : couleur différente selon l'état — cohérent avec le style déjà utilisé pour le bandeau "Voyage débloqué" existant, avec une variante visuelle distincte pour l'état verrouillé).
- Le bouton unique doit être suffisamment explicite pour qu'il n'y ait aucune ambiguïté sur l'action qu'il va déclencher (éviter un simple pictogramme sans texte).
